Java 图像缩放方案

2011年02月18日

  java对图片的操作功能可谓是比较弱，而且效果比较差。而在项目中，又经常会对图片进行操作，比如：用户上传一个图片上来做为头像，系统就要对图片进行缩放，以使头像符合界面显示要求，并且减少头像文件大小，对站点的响应有一定作用。

  图片绽放，精度肯定会损失，一般我们都要求缩放后的图片在视觉上平滑，不要出现锯齿，下面列举一些实现方案：

  一、利用j2se的Graphics、JPEGImageEncoder等类实现图片缩放，这样的代码网上很多，有兴趣的可以google一下。这种方案的缺点是：效果不理想。还有一点，JPEGImageEncoder在有些jdk实现版本中不支持，比如：OpenJDK。

  二、借助于开源图像处理框架Java Image Filters，网址在这：http://www.jhlabs.com/ip/filters/index.html，该框架可以实现常见的图片处理，在http://www.oschina.net/bbs/thread/8248还有一篇文章，介绍如何使用。不过，你打开源代码看看图片缩放的源码，就会发现框架也是简单的封装了j2se的图像处理api，因些，缩放效果很不理想。当然，该框架中的其它图像处理功能比较强大。

  三、使用 ImageMagick,这也是一个开源框架，c语言写的，还有专门的java的接口，在http://www.imagemagick.org /script/api.php可以看到ImageMagick的各种语言接口实现。该框架的功能非常强大，图像处理后出来的效果是最好的，像是缺点是：环境搭建困难，比如：在windows下需要dll,jar包的支持，如果是linux或unix系统要手动编译.so的库文件，新手搞起来肯定比较困难。

Java Mis类型项目招聘技术点总结

2011年01月20日

  我有过一些在软件公司招聘的经历，下面谈谈在招聘过程中我所关注的一些地方，做为新手可以从这些方面去掂量下下自己，找一个合适的方向。

  当然，不同的公司，招聘条件并不相同，不同的项目，招聘条件也不同。但是也存在一些招聘误区，比如：有的公司喜欢作秀，把题目出得很难很深，这些考试内容在实际项目中根本用不上；也有的公司把自己花了很大精力才解决的问题拿来考面试者；还有的公司出一堆IQ测试题等等，这样的情况非常多。我还是喜欢务实一些的，工作中、项目中用什么就考什么。据我所知，成都的很多公司喜欢来虚的，深圳、上海等地的公司更务实些，呵呵。

  我以前出的一些题目，主要是从实际项目出发，考核应聘者的一些基本能力，不同类型的项目，条件也不一样，但有很多基础的东西是必须要掌握的，下面以开java开发MIS系统为例，进行讲解。

  1、java基础

  a、基本语法：顺序、分支、循环这三种基本的程序结构要了解，同时要关注一下执行效率问题(比如：变量的申明范围、类型等)。

  b、对象的继承、封装、多态，对象修饰符，构造函数等知识。

  c、集合的使用：数组，set，map，list的基本用法。

  d、基本流操作：InputStream,OutputStream及几个常用子类，比如：操作文件的、操作带缓冲区的、带整行读取功能的、操作字节的等，流操作虽然用得不多，但有时还是要用的哦。

  e、多线程原理。对理解服务器请求处理及资源同步有帮助。

  f、工具类的使用：有这么几个类很常用SimpleDateFormat，Calendar，Math，Radom，字符串与数字类型的转换，以及正则达式类。其实也就是不同类型的转换、格式化等操作。

  2、web部分

  a、socket的原理，http头（很重要）。这些东西在实际编码时几乎不用，但又需要程序员理解，因为这些知识对解决实际问题非常有帮助。

  b、4个web作用域，必须要有一定的认识。session与cookie的联系区别要清楚。

  c、页面与业务的交互流程要清楚，还要了解一些安全交互方面的知道，比如：数据验证，跨域等。

  d、el和jstl，jstl中的core，fmt和fn都要熟悉，特别是日期、数字的格式化。

  e、对xml解析方式要清楚，有的项目在这方面要求还比较高。

  3、页面展现

  a、基本html元素必须熟悉。

  b、合理使用css定制页面外观。

  c、基本的js操作方法（取页面元素、加载事件、触发事件等）。

  e、一些js框架，比如：jQuery，Extjs等，其中jQuery是重点哦。

  f、父、子页面（或iframe）间的参数传递及互访。

  g、ajax交互原理。

  3、数据库方面

  a、基本的sql语法，特别是子查询、分组，聚合、分页。

  b、特定平台的存储过程（有的项目需要）。

  c、范式的基本认识。

  d、一些性能方面的知识。

  4、框架部分知识

  a、struts1.x，重点是一个交互流程。DispatchAction，FormBean的基本用法。

  b、hibernate,关系映射的配置是一个方面，另一个是HQL的写法，特别是关联对象，多对多对象的hql写法。hibernate性能方面要有所了解。

  c、对spring容器获取的方法，特别是在web环境下的获取。对spring的AOP、数据库事务管理（重要）以及bean的生命周期要有一定了解。其实spring本身的配置和使用还是相当简单的，重要的是原理。

  e、对其它框架要了解，比如：struts2，ibatis,ejb3,webservice相关框架等等。

  5、架构及设计模式

  架构方面对于一个普通程序员，要求不是太高，但常见的mvc、失血模型、事务脚本、dao这几个，一定要理解透彻

  设计模式只做一般了解，工厂、单例、装饰、适配器、观察者、门面 这几个模式相对来说比较重要，因为模式体现了面向对象设计原则，对代码重构及理解其它写人的代码都有好处。

  上面的这些考核知识点比较散，也不完善、不成体系，但却是项目中所需要的，是从实战出发所总结，大家要引起重视。

  最后，还有两点也是一个公司比较看重要的：

  1、人品，所谓“有德有才 破格重用，有德无才 培养使用，有才无德 限制录用，无德无才是坚决不用”，说的就是德，可见，人的品德是多么的重要。

  2、对技术的敏感性。

  我曾经面试过一个法律专业的小伙来公司做技术，面试时就发现该人非常踏实，诚实，人比较聪明，此人已经自学了一些编程知识，但当时老板不太同意，最终还是招进了公司。这个小伙子我只带了一个月，就很快能够独顶一面，进步得非常快。也许有的人天生就是吃这碗饭的。

  软件行业中对技术还比较迷茫的人，希望读完本文后，能给你点启示，毕竟过来人都走过一些弯路的，这些总结对你一定有用。

网页解析利器——jsoup

2011年01月19日

  从第三方网站上抓取一张页面，分析其中的内容，然后转换为自己的格式并持久化，这就是网络爬虫的操作步骤。目前，对网页的抓取及解析也在互联网项目中有较为广泛的应用，下面简单谈谈一些网页解析工具的使用方法。

  网页的爬取及解析，应该利用一现有的工具或框架去实现，而这些框架的选择，至少要考虑下面的一些因素：

  1、应该具有一定的兼容性，比如：原始页面中的一些语法错误、未关闭标签等，应该能够被正确解析；

  2、应该可以方便地查找、过滤页面元素；

  3、应该要以对找到的元素进行操作，比如：修改，删除等，当然也可以加入新的元素。

  我主要说两个常用的网页解析工具，一个是htmlParser，这个框架出来得比较早，并且有比较广泛的应用，官方站点在http://htmlparser.sourceforge.net/，下面是一段示例代码：

  上面将得到页面中的所有图片链接。其中的extractAllNodes方法有很多种实现，可以按元素类型、包含的属性等等条件进行过滤，我以前的一些项目中就曾使用这种方法来做，也用得十分的恼火，让人头疼的地方主要是：

  1、容错性差，目标页面有错误，解析起来就得不到正确的结果，为此，当时都研究过firefox的解析包，但发现十分复杂，放弃了（注：如果每个页面都通过xhtml标准检查，那么用dom4j中的xpath就可以解决网页解析的问题了，呵呵）。

  2、要仔细分析目标网页，针对目标网页的页面元素层级关系，编写相应的解析代码，写出的代码非常冗长（主要是元素查询、过滤很麻烦），可维护性不高。一旦目标网站页面格式调整，再次调整代码，修改解析代码非常痛苦。

  3、页面的修改、重组都不方便。

  用HtmlParser的经历可以说是比较痛苦，但在当时的情况下，HtmlParser是最好的一个java开源网页解析器，也只有将应着用了。同时也期待着新的框架问世。

  直到有一天，发现了jsoup，以前困扰过我的问题终于得到了解决，并且解决得很彻底，是到了跟HtmlParser说再见的时候了。jsoup官网地址在http://jsoup.org/，整个jar包135kb，非常小巧，而且不依赖其它包，够强悍。下面贴出一段示例代码（具体使用官网上有Cookbook，另外chinaos版主也写了一篇jsoup的使用文档，自己去google一下吧）：

一个复杂的定时任务架构原型实现

2011年01月11日

  我在前面的文章中讲过定时任务在项目中的应用 —【项目中定时任务实现方式比较】，并且我推荐大家利用操作系统的定时任务去完成相应的功能，对于简单的定时任务，操作系统的定时任务功能完全没有问题的，但是，如果项目需求复杂一些，操作系统的定时任务恐怕就不能胜任了。最近，我们在重庆移动实时监控项目中就遇到了这个问题，下面贴出界面原型，并简单描述该部分的功能需求。

  实时监控系统中可以建立多个预警任务，每个预期任务中包含一个运行周期，运行周期从周期类型和天内频率两个方面来描述这个任务的执行触发条件，如下图所示：

运行周期界面原型图

  每个预警任务建立完成后，还可以暂时、启动任务，或立即执行任务，并且预警任务的运行周期是可以修改的。预期任务列表界面原型如下图。

预警任务列表图

  预警任务在达到运行周期中设定的触发条件后就会自动运行，在运行过程中，系统将向预警任务中设计的目标网元、数据源等监控目标进行通讯，并采集性能数据，经过整理后将结果存入每个预期任务对应的预警结果中。

  预警任务可以由人工单独运行（即：立即执行一次），或者按照设计的运行周期自动触发。

  很显示，在这个需求中，用操作系统的定时任务已经无法达到我们要求，原因是：每个预期任务的运行同期都不相同，而且比较复杂，并且预期任务可以随时修改，周期也可以修改，我们不可能在这些东西修改后，再去修改操作系统的配置文件。

  根据预警任务这一块的需求，我们在项目之初设计了架构原型，并写出了相应的代码，实现思路是在项目中深度整合quartz框架。架构原型设计图如下：
