  九、项目中的决择

  ibatis是一个“半自动化”的框架，使用起来自然没有“全自动化”的方便，也正是因为“半自动化”的原因，ibatis才拥有自己独特的适用场合和表演舞台。

  在项目中进行底层技术选型时，我认为有存在下面的因素可优先选择ibatis：在遗留系统现在数据基础上进行开发；公司开发人员非常熟悉且喜欢写sql （有些老程序员更习惯以数据库表为中心的开发模式）；项目的其它非功能性需求。除些之外，请选择更加OO的ＯＲＭ映射框架。

  我曾经看过好几个公司ibatis开发的项目，对象和表结构都是一一对应（更有甚者把表外键都建成了对象属性），对象关系几乎不存在，也就没有了领域建模的概念。从我上面写的一些配置代码也可以看出，ibatis在处理一对一、一对多、多对多查询（主要是N+1）上都这么复杂，更不用说一对一、一对多、多对多的增、删、改操作了（请想想hibernate中的cascade，inverse带来便捷），也就是说，对象之间的所有关系要程序员手工书写sql去维护，这对于增、删、改操作还是非常麻烦的，一些级联操作还得在业务实现代码中主动调用，增加了业务的复杂性。

  也许，在ibatis的项目中将对象和表结构进行一一映射是一种最方便、最快捷的方法，但对于喜欢OO建模的人来说，可能无法接受。项目中如果用表结构对象代替领域对象，只会增加项目实现复杂性和后期维护成本。

  话又说回来，Hibernate、JDO这些ORM框架可以较好地支持对象透明持久化，但学习成本相对来说也要高一些，也正是这些原因让很多项目经理放弃了这些技术。还有人说hibernate性能低下，项目失败的风险很大，我认为这些人并没有全面、深入地掌握Hibernate，比如：对于性能低下的部分，你可以通过hibernate执行native sql解决，甚至存储过程。还有一个是技术转型的原因，对于从C/S转到B/S的老程序员来说，我很清楚OO思想和以表结构为中心的思想之间的冲突，但只你要突破了，你才会达到另一个认识层次，才能体会到其中的好处。

如何处理项目中的数据验证

2011年06月16日

  有人说，所有的程序本质上都是“输入->处理->输出”，仔细想来确实如此。话又说回来，你输入的是垃圾，输出的肯定也是垃圾，你输入的是“炸 弹”，面临的将是系统崩溃、瘫痪，也就无法得到预期的结果。因此，数据的验证，在任何一个项目中，都是不可回避、不可忽视的问题。

  下面，我还是按web应用程序的分层结构和大家探讨一下数据的验证实现的一些问题。

  一、表现层： 表现层是接受客户数据输入的来源，第一道数据检验“关卡”应该也在这里。

  A、 客户端的验证 web系统客户端采用的方法是：用js对客户在浏览器中输入的数据进行验证，这些验证通常不包括业务方面的内容，比如：字段是否已经填写，字段是否为数字，字段是否为email，字段长度是否满足要求，是否是一个合法的身份证号码，是否是合法的日期等，如果结合正则表达式，则可以实现一些更为复杂的数据验证功能。

  客户端有时还得借助ajax实现一些业务验证，比哪：用户注册功能中，就可以利用ajax及时检测用户输入的登录名是否已经重复等，这类型的验证已经具有了业务逻辑的影子，类似这种情况的简单业务验证是允许在客户端验证中出现的。

  客户端验证通常采用一些验证框架提高开发效率，比如jQuery中就有一些很不错的验证插件，不仅验证条件的书写很简单，而且验证没有通过的提示信息更加的友好。我推荐jQuery validate，这个插件的使用我会在另一文件中写出。

  当然，客户端的js验证可以被某些不怀好意的人轻易绕过（比如禁用js或用特殊工具提交），因此，真正的验证还得在服务器上做，客户端的验证，就全当做为提高系统人机交互效率而存在。 如果你将flex，java swing这类技术应于于表现层，原理同上，我不再复述。

  B、服务器端的验证 提到表现层，MVC模式肯定要说说。我们知道，几乎所有的MVC框架，都提供了数据验证实现，这种数据验证功能是在服务器端实现的（其实就是服务器端的验证了），因些它不会被恶意绕过。 validate验证框架的原理是把通用的验证逻辑代码提取出来，将应用程序界面输入字段元素的验证规则写到配置文件（xml）中去，并由框架根据配置的验证规则调用验证逻辑完成验证，如果验证不通过，将跳转到指定的出错显示页面，控制器（controller，即action）中的业务调用代码将不会被执行。

  使用验证框架的优点是统一、规范，程序员手工验证代码移走后（变成了通用的验证代码），控制器中的代码将变得非常的简洁，这有利于系统维护。当然，缺点是引入了新的框架，系统和验证框架产生了耦合（不过，这点对于大型项目来说并不存在问题）。

  我认为，小系统或者表单验证字段不多的情况下，没有必要引入验证框架，程序员直接在 controller中加入验证代码即可，即手工书写if else，或者你也可以把你的验证代码重构一下，提出一个工具类来。

  如果一个表单中验证段过多，这种方法就不是太好。自己写验证代码还有一个缺点是数据回填问题（就是将不合法信息提示给用户的同时，用户以前在页面上输入的数据都要回填显示，以避免用户从新来过）比较麻烦，需要程序员专门来处理这种情况（其实，不同的MVC框架也是借助于自定义页面标记来实现的）。如果有了客户端的验证，数据回填显示似乎并不存在问题，呵呵。

  二、业务层：

业务层的数据验证，主要是针对输入数据在业务逻辑中是否存在问题，也就是说，表现层和业务层的数据验证，应该各自有各自的职责，表现层偏向于数据是否为空等简单的判断，几乎不会和数据库发生交互，而业务层主要是针对业务逻辑的数据合法合理性判断。

  说起来简单，做起来并不容易。在实际的项目中，我们很有可能将一些业务数据合理性检测放到controller中去做了。

  举个例子：新用户注册功能中，需要检测用户名是否重复，而检测用户名是否重复要和数据库交互，我看见很多人都是在controller中去调用service中一个叫checkUniqueRegUserName的方法，如果重复将直接跳转出错页面。这种做法个人认为不是太好，检测用户名是否重复属于业务范畴。

  如果你把这个检测过程封装到保存注册用户方法中，通过异常机制来检测注册过程中发生的故障，也许更合理些。你可以定义这么一个方法 public void register(User user) thorws RegException。

  有了这个方法，我们的程序代码结构将更加的清晰。页面表单输入的User数据合法性检测由验证框架或手写if else来向用户报告，User的名称是否重复由register方法向调用者报告，职责更加明确了，也更符合“职责单一原则”。

  当然，哪些是普通合法性验证？哪些是业务逻辑验证？对这两者的划分不同，你的验证代码在服务器端写的代码将不一样，我认为只要和数据库交互或和其它领域对象协作才能得到的结果都是属业务逻辑验证。

  三、持久层

  持久层也可以对数据进行验证，有两个验证地点，一个是在ORM框架中，二是用数据库自身的约束机制。

  ORM映射框架，比如说hiberante，就可以配置column的字段类型，是否可以为空，长度是多少等，如果在保存或修改的过程中，数据不符合hbm配置要求，hibernate框架会发现这个错误。 数据库中可以通过设计字段类型，是否可以为空、外键、主键等，通过数据库自身的约束机制来拒绝非法的数据。

  持久层的数据验证已经属于最终的数据验证了，我们的数据验证策略应该是：尽量提前验证。如果你可以确保前阶段数据已经经过了严格的验证，持久层这边可以安全放行，不做检测。 经常有人问我项目中数据库这边的外键约束，非空约束是否要加上，我一般采用的策略是项目开发阶段不加（给程序员开发带来方便），项目上线后根据情况来看，加上是最好的了（可访止DBA误操作引入非法数据）。

  总结：

  1、 客户端用js框架帮助我们实现验证；

  2、 服务器端通过MVC验证框架帮助我们实现（请看我的《都spring mvc了（二）》，spring mvc 是如何实现的验证的）；

  3、 分清哪些由业务方法去做的验证，哪些是由普通的验证；如果前面做好了，持久层验证可以省去。

省质监局一期项目结束了，总结一下！

  经过两个半月的努力，省质监局一期项目告一段落，在这段不长的时间内，我们开发团队成员已变得成熟、稳重，能够从容应对开发中遇到的棘手问题，这点我感到非常欣慰。

  省质监局项目是老大借助些关系拿过来的，这个项目开始是由另一个公司开发的，但最终结果不被质监局上层领导们认可，当我们接手这个项目时，已经有一些可供参考的界面原型，但当我们向该公司要更多项目资料时，对方不予积极配合，给的PD物理模型上都有很多错误，很显然这不是最终的东西（据说这个模型他们反复讨论了两个月才定下来的，肯定不会轻易给我们的）。

  根省质监局领导协调，指示我们可以全新设计，有这句话就好办多了。

  项目开始后，我们把对方遗留（未完成）系统页面进行了截图，拿回来熟悉业务流程。在梳理业务流程的过程中，我们产生了很多疑问，并且也发现了对方系统中存在的一些不合理的操作，对一些不清楚的流程，我们先进行“自我假设”，假想合理的业务流程，因为第二天要到客户现场进行需求的确认。

  当天会议结束，我整理了10个有疑问的问题清单，第二天和开发小组成员一起来到现场。想见到了省质监局信息办主任来确认需求还有点困难，领导们大多都很忙，一上午连续开了两个会，快到中午了才些时间，我带了一个开发组长来到主任办公室，对有问题的需求进行了一一确认，并且得到了他们的一些业务文档，当然，最重要的文档是《全省质监系统信息化建设和应用考评实施方案》，这个文件详细描述了很多重要的业务规则及内容。

  开发的详细流程可以参阅我的博文《一个迭代周期内的项目开发流程》以及《一个简单的j2ee入门教程（第一部分——需求分析 ）》至《一个简单的j2ee入门教程（第六部分—细节完善及测试）》部分的描述。先期的《可行性分析报告》、《项目计划书》等用模板写好即可，尹哥写的（尹哥那还做了一个甘特图，感觉不错）。

  开发人员花了大概3天的时间进行原型的设计，之后我们一起进行了评审（注意，用Balsamiq Mockups做的原型只可用于内部评审，一般的客户无法理解这种原型草图）。原型确定下来后，程序员分工进行页面的实现，美工进行首页、主页的制作及页面风格的确定，开发人员进行具体页面的实现，4天时间完成了所有的页面。

  项目小组带着页面到客户现场，进行演示和确认，这个过程大概持续了一周的时候，每次需求演示都会带回来一些新的需求及改动，因为这段时间我比较忙，一直没抽出时间过去，所有的工作都由开发组组长负责完成。每次演示回来，我都要和开发组成员一起对改动进行评估，主要是考虑后期的可实现性及操作的合理性。（尹哥这个时候已经全身心做Android 项目了，我们这边已无法顾及）。

  需求分析总共花费了近1个月的时间，这个阶段一定要慢一些，需求搞不清楚对后面的影响就大了。话又说回来，做过政府项目的人都知道，政府部门的需求是变化最快的，各个领导的意见都有分岐，以至于我们每次过去都会有些新的东西要添加，有时甚至会和头一天的说法自相矛盾，但这些都没有办法，我们面对的都是些领导。

  当我们认为html版的原型基本得到客户确认（只能是基本确认，我们要就此打住，否则没完没了，无法进行后期的开发了，呵呵），这个时候发现已经在原版本的基础上添加了很多新的东西。

  后来安排项目组写出了《需求分析说明书》，按功能模块的方式描述了每个模块的功能及操作细节、业务约束（尹哥喜欢按这种需求描述方式，我个人更偏向于用例文本），这些文件最终都要向省质监局提交。

  接着进入设计阶段，由于我们已经有成型的技术框架，因此，架构原型上的要做的东西很少，我只做了一个webservice的界构原型，主要是对安全性进行了验证（请参阅我的《webservice之cxf实现》及《webservice架构设计》）。领域模型的完善、物理模型的制作是设计阶段要做的第一件事，这些东西我安排项目组的人员去做，最终由我审核后通过。

  接下来，开发人员设计了业务类及接口，参照已经成型的界面原型应该不成问题，经过审核后通过。设计工作历时2天时间（其实可以算是详细设计）。

  终于可以进行项目代码实现了，我安排一个开发人员完善hbm文件配置，其它人员搭建项目结构、配置svn以及页面的转换等等前期工作，我制定了近1周的工作任务（后续的开发都是以周为单分配），工作任务以“一个CRUD操作1.5工作日”为标准确定时间进度，当然，安排任务时要考虑到业务代码实现的复杂性以及开发人员的技术能力，因为我们开发组的成员都不能算是熟练的开发人员，很多技术都需要在实际项目中磨练，在任务分配时要考虑很多的冗余，一般项目周期的是10%~20%。

  后期都是驻客户现场项目开发，偶尔有些问题，直接咨询客户，一般的技术问题都由我电话解答，QQ解决，电话说不清楚的或者一些重大问题我都会将开发人员召回，开会明确。这一段时间，我们的开发人员非常的辛苦，工作时都是只休息一天，我经常在周六接到他们的电话，问题解决之余我都要叮嘱一下“注意休息”。有的时候项目组长解决一个问题会坚持到晚上1、2点钟，开发小组群中很晚都还有人在“冒泡”，我也心存感激。由于我这边的工作性质，开发小组的人员回来，我一般都没有充分的时候沟通，只有等到下班后，再集中解决遇到的问题，我们的开发人员确实不容易。其实，我能做的只是提供完善的技术支持、解决方案，并对开发过程进行把控，排除影响进度的因素。

  近20天的开发终于结束了（中途我进行过一次代码评审），大家也比较疲惫，项目也可以给客户演示了。

  由于项目没有经过严格的测试，第一次演示失败，流程走了约2/3时出现故障无法进行下去，后面又经过2天的bug修改及测试。

  第二次演示基本通过，但客户随即提出了新的需求变动方案，拿出了一个我们之前从没有见过的文件，这个文件将会对代码及页面交互实现代码造成很大的影响（还好，设计上不需要变化），我也很清楚开发人员要改动的地方，这次需求变更的提出对开发人员的打击很大，回来开会时个个都垂头丧气。尹哥也刚出差回来，我们只有安抚开发人员，给他们讲需求变动是项目开发中常见的事。我们当即和对方的一个负责人协商，并预估了这次变更实现需要的时间，希望对方做出让步，我也让开发人员先回去好好休息一下，第二天再说。经过第二天的现场协商，对方做了让步，我们双方选择了一种折中的方案，最后，经过近一周的修改，完成了这次需求的改动。

  第三次演示也通过，在演示过程中，我们的开发人员曾经怀疑我们设计上的一些问题，没想到对方负责人说：“你们的设计是对的，错不了”，还说我们的开发人员有些问题比他想得更深入、全面，呵呵，确实，这个系统做完后，我们项目组的人都变成了这个领域的业务专家。

  第四次演示给其它更高层领导看，这次演示基本没有大的问题，仅仅对主页面中的一些显示效果提出了修改意见。

  上周，我们完成了项目在质监督局服务器上的布署（32G内存的服务器，真够有钱的），听客户说这一阶段的项目暂不投入实际运行，他们想要加入更多的一些东西，想让这个系统变成更加的通用，并想让我们在一周的时候内实现这一部分需求添加。我预估计了时间，光是这项改动，项目组在一周内肯定是完不成的，更何况，本期项目组的开发人员要陆续离开（这点确实有点头痛），新开发人员的项目交接也需要一段时间，这些新的东西，恐怕只有在二期中完成了。

  目前，项目二期工程即将启动，项目组人员的搭建、培训以及新旧人员顺利交接都是近期的工作（项目文档这些还算完善）。新班子建好后将进行二期的需求调研，希望我们二期项目的开发可以走得更顺一些。

  质监局项目一期开发过程中，我体会很深的有下面几点：

  一、项目中能用旧技术、方案解决的，最好不要引入新的东西。

  本项目中有一个“考评方案维护”模块，尹哥从产品交互效果、项目亮点等方面考虑引入了TableTree控件（一个js控件），这个控件的引入给项目后期的实现带来了困难，主要表现在该控件api不全，甚至缺少我们要的一些特殊处理的方法，最后在这一块耗费了一个程序员近三周的开发时间（还有其它人帮忙），边写功能还要边读 果js框架源代码。

  最终做出来的效果是非常不错，但问题又接出现了，这个模块的交互全部用js实现（包括ajax文件上传等复杂操作），致使代码量非常庞大，核心操作部分js代码近1000行，虽说这块代码写了详细注释，但目前阅读、修改起来都非常困难（包括代码实现程序员本人）。

  因些，另一个js弹窗控件（可遮罩iframe以外的地方）也准备引入，最后我还是决定放弃这个控件，而改用普通的请求响应交互模式完成，因为我很清楚这个控件对开发带来的负面影响。

  我的原则是：先实现功能，再考虑交互效果。

  二、客户就是上帝，需求变化无常，要尽量做到设计上的可扩展性。政府单位的项目的开发更是要注意这点，这要求我们在设计上有一定的前瞻性。本次一期项目开发中虽有些需求变化，但对整个项目设计并没产生影响，仅仅是对程序员的代码实现有很多改动。

  当然，预见未来的变化只是一个相对的说法，如果大家都能做到超前预见，那么一款软件的生命周期可能会大大延长，很多程序员也就要“下岗”了。

  质监局项目二期的需求中，我们就没有预见到，信息办主任也从未提及二期的规划和设想，现在算是明白他的想法了：完全抛弃一期项目，二期全新从头开发（政府单位真的这么有钱？）。难道我们两个半月的努力白费了？我想，我们还是得要在一期项目上更改，只不过有些地方要动动“手术”，影响还是有些大，但不至于全盘推翻。

  三、项目的开发流程我做了些小调整。PD中的ER模型省去了，直接做物理模型。ER和UML工具中做的领域相比并无优势，因此，我们的领域对象、业务接口这些将直接从UML工具中出图出代码 ，建表 sql、数据字典直接从PD物理模型中出，省去ER会节省时间。

  四、ORM关系映射文件要选择有经验的开发人员来完成。本项目中我安排给了一个普通程序员来做，致使在项目进行的全过程中，不断发现映射文件中的配置小错误，这点对项目的影响非常大，浪费了我们很多宝贵时间（有些问题查来查去最终发现问题在映射文件上，大家只能埋怨写映射文件的程序员）。

  五、 webservice 接口的数据交互方式，我们采用的是由第三方系统主动推送，这种方式在项目中并不常见。我当时提的其它几个方案都被信息办主任否定了(请查看我的《你一定要知道的几种项目间数据交互方式》)，呵呵。后来才知道，此人计算机专业出生，10年前是做开发的，对我们这行了解很多。所以，以后做需求调研的时候，最好还是要把对方的背景了解清楚，做到“知己知彼”。

Hibernate中的“隐喻”

2011年06月10日

  隐喻，百科上的解释是“隐喻是在彼类事物的暗示之下感知、体验、想象、理解、谈论此类事物的心理行为、语言行为和文化行为。”，隐喻一词用在计算机软件领域，意思是：我们想当然或我们认为是这样的，但实际的执行情况却大相径庭，直至错误出现才意识到问题严重性，下面我举两个实例来说明。

  第一个例子和hibernate继承映射（多态）有关，比如：我们的领域中有员工和客户两个对象，每个员工和客户都有自己的附件资料，是一对多双向关联关系。由于附件的大多数属性都相同，我们使用了泛化关系，即：员工附件、客户附件都是附件类的子类型，这种设计方式在项目中比较常见，领域模型见下图。

  上面的这个案例发生在一个真实的项目中（不过为了说明问题，代码是我另写的一个演示），程序员一直在给我抱怨查出的数据条数不对，但有时又是正确的，我当时没有在意，让他们检查测试数据是否存在问题。后来项目要上线了，这块代码开始报子类类型转换出错，看来不解决这个问题肯定是不行的了，逐坐下来和其中一个程序员一起找问题原因所在，发现了这个hibernate中存在的“隐喻”。解决办法也很简单：在set集合的配置上加上where属性再次指定子类类型字段标志值，比如在employee映射文件的set上加上 where = “f_accessory_type=1”，问题终于解决了（其实也可以通过更改Customer、Employee的OID生成方式为uuid解决本问题）。

  第二个例子和hibernate的hql查询有关。比如Customer和Order是一对多双向关联，关系图如下：

客户与订单领域关系图

  客户中包含订单的集合，而该集合默认情况下是lazy的，但有时为了提高性能，我们想用一条sql加载用户及其订单，我们可以写出下面的HQL。

  from Customer c left join fetch c.orders o;

  但这条HQL得到了客户List集合中，却有很多重复的用户对象，又和我们的想法不太一样，我们只是想加载客户时初始化客户的订单集合，实际上我们得到了一些重复客户对象集合。

  正确的HQL应该这么写：select distinct( c ) from Customer c left join fetch c.orders o，这条HQL会得到唯一的客户集合（我见有人写过用Set来过滤List集合重复元素的代码，其实没有这个必要）。

  补充一点HQL的东西，很多人是写不出来这条HQL的：查询订了iphone订单的客户集合，HQL可以这么写：select distinct( c ) from Customer c left join fetch c.orders o where o.name = ‘iphone’，你不要老是想着orders是集合而觉不能用o.name这样的条件哦，呵呵。这种查询同样适用于多对多中集合条件的查询。

  记得Ruby on rails中曾经提到一个“惯例重于配置”的原则，也就是说很多规则不用配置，会按我们的惯例进行处理，但很多时候，恰恰是我们认为惯例是这样的，而实际上却是另外的“一番景象”。

  “实践是检验真理的惟一标准”，软件开发人员也应牢记这一点！
