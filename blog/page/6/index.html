  上面的这个案例发生在一个真实的项目中（不过为了说明问题，代码是我另写的一个演示），程序员一直在给我抱怨查出的数据条数不对，但有时又是正确的，我当时没有在意，让他们检查测试数据是否存在问题。后来项目要上线了，这块代码开始报子类类型转换出错，看来不解决这个问题肯定是不行的了，逐坐下来和其中一个程序员一起找问题原因所在，发现了这个hibernate中存在的“隐喻”。解决办法也很简单：在set集合的配置上加上where属性再次指定子类类型字段标志值，比如在employee映射文件的set上加上 where = “f_accessory_type=1”，问题终于解决了（其实也可以通过更改Customer、Employee的OID生成方式为uuid解决本问题）。

  第二个例子和hibernate的hql查询有关。比如Customer和Order是一对多双向关联，关系图如下：

  客户中包含订单的集合，而该集合默认情况下是lazy的，但有时为了提高性能，我们想用一条sql加载用户及其订单，我们可以写出下面的HQL。

  from Customer c left join fetch c.orders o;

  但这条HQL得到了客户List集合中，却有很多重复的用户对象，又和我们的想法不太一样，我们只是想加载客户时初始化客户的订单集合，实际上我们得到了一些重复客户对象集合。

  正确的HQL应该这么写：select distinct( c ) from Customer c left join fetch c.orders o，这条HQL会得到唯一的客户集合（我见有人写过用Set来过滤List集合重复元素的代码，其实没有这个必要）。

  补充一点HQL的东西，很多人是写不出来这条HQL的：查询订了iphone订单的客户集合，HQL可以这么写：select distinct( c ) from Customer c left join fetch c.orders o where o.name = ‘iphone’，你不要老是想着orders是集合而觉不能用o.name这样的条件哦，呵呵。这种查询同样适用于多对多中集合条件的查询。

  记得Ruby on rails中曾经提到一个“惯例重于配置”的原则，也就是说很多规则不用配置，会按我们的惯例进行处理，但很多时候，恰恰是我们认为惯例是这样的，而实际上却是另外的“一番景象”。

  “实践是检验真理的惟一标准”，软件开发人员也应牢记这一点！

不得不说spring事务管理的几个问题

2011年06月09日

  首先说明，不是spring的问题，是程序员编程时要注意的问题，不要被标题误导了哈。 spring通过aop机制为项目的事务管理带来了极大的方便，有了事务切面，我们的代码中再也看到commit、rollback之类的语句，我们要做的仅仅是在需要事务的地方声明事务的一些属性，spring给我们的项目带来了实实在在的好处。

一、事务声明位置

我们先来看事务声明的位置，事务声明的位置无外乎在表现层，业务层，持久层这三个地方。 如果将事务声明在持久层，粒度会显得过细，因为持久层一般都是些普通的CRUD操作，大多方法中仅仅一条sql语句而已；事务可以放在业务层，一个业务方法一般情况下会作为一个完整的事务，所以，事务控制在业务层是一种通用的做法；也可以将事务放在表现层的请求执行的方法上，当然这种情况并不多见。

二、事务属性

spring事务属性配置项包括：事务传播属性，事务隔离级别，只读提示以及事务超时时间属性，这四种属性中，我们关注最多的是事务隔离级别和只读提示，其它两种直接使用系统默认值即可。这些所有的可配置项，可以在spring的org.springframework.transaction. TransactionDefinition源码中看到，源码节选如下，英文好的可以研究一下里面的说明：

  事务传播属性，一般选用PROPAGATION_REQUIRED，表示此方法的运行需要运行在一个事务环境中，如果没有就创建一个新的，这种方式适合大多数业务情况。如果你不向让被调用的方法事务失败影响到本方法的事务，可选用PROPAGATION_REQUIRES_NEW。

  我要强调的是，事务传播属性用于不同的类之间的业务方法调用中的事务的控制，一定要注意是不同类中不同业务方法，同一个类中不同方法可能不起作用，这是由spring aop方法切入实现有关(同一个类中不同方法之间的调用将不被拦截)。

  事务只读提示readOnly属性，标明了该事务不可修改数据，一般查询方法都标成readOnly，可以提高系统性能。

  事务隔级别是本方法执行过程中，数据库在事务方面应采用的级别，一般采用默认值ISOLATION_READ_COMMITTED即可满足要求，因此，该属性我们一般不设置。

  事务超时属性一般不设置，用默认值(不限制)即可。

三、事务的提交与回滚

  spring通过业务方法执行过程是否抛出了异常来判断是回滚还是提交事务，看似简单的一句话，却被很多人忽视了一个非常重要的一点。那就是spring只检测RuntimeException及其子类，对检测到的Exception将不回滚，因些，在事务属性配置中有一个rollbackFor属性，用来指定哪种异常回滚。对这点理解不清会得到惨重的教训的，呵呵。

四、编程式事务、配置式事务，xml or annotation?

  编程式事务可以在小型项目中使用，可以使用TransactionTemplate的doInTransaction回调实现，要回滚事务需要设置transactionStatus.的setRollbackOnly来设置事务状态；也可以得到一个PlatformTransactionManager实例才操作，具体请参阅Spring参考手册9.6.2节内容。

  配置式事务是强烈推荐的一种方式，但是配置式事务的属性可以写到xml文件中，这种方式对业务方法（假设事务控制在业务层）的命名有所限制，也就是要求业务方法命名要有规律可循。如果你采用annotation来配置，则没了这个约束，这也更加有利于业务方法的设计（业务方法命名一般更具有业务含义，天马行空也许更适合些）。

  总之，事务管理是做软件系统的程序要不可回避的问题，用spring帮助我们来管理事务是明智之举，上面的这些问题希望对你处理好项目中的事务有所帮助。

言娃娃一岁了

2011年06月08日

  今天是6月8号，言娃娃来到这个世界上已经整整一年了。早就喊言娃娃妈写一篇生产日记，但到今天也没有看见到，只好由我来记录一下这个流水账了，呵呵。

  6月7号，早晨起来发现有生产前兆，我就向公司请假，收拾了要带的东西，包括出生后的包被，小衣服，奶瓶，奶粉、尿不湿这些，最重要还有两块巧克力和一罐红牛，言言妈在网上学的说在生产过程中需要补充些能量。就这样，我们出发了，就我和言言妈两个人，真有奔赴战场去打仗的感觉，心里面没有底，不知道要发生什么事情。两个人提着东西走在小区门口的路上，孤孤零零的好可怜哦（言娃娃奶奶在家守房，因为还不知道当天是否真的要生产了）。

  打taxi到省妇幼，先到普通门诊，描述了情况，医生说：你还不去挂急诊，在这等什么呢？后来，急诊科医生检查后，马上开了住院证明。要住院了，妈哦，看来真的是要准备战斗了。

  我去交了5000住院费，到了四楼住院部，被告知没有床位了，我们不仅感叹中国的人真多啊。从护士站上领了卫生盆、护垫这些东西，最后在四楼西边的走廊尽头安装了一张，言娃娃妈妈才能躺下休息一会。当时觉得我们好穷哦，要是有钱，肯定去安琪儿医院生产了，谁让我们是普通老百姓，真的委屈言娃娃妈了。

  我隔一阵就往护士站跑一次，问问有没有新的床位出来（后来才知道，根本不用你去问，医院是按顺序排的，很公平）。护士也是每过2个小时来检查一次，我的心也放下来了很多。这时言娃娃奶奶和舅舅也赶到了医院，中间言言妈的一个导游同事还过来看了我们，买了一套小孩的衣服（因为是个女同事，她和言妈妈一样喜欢女孩子，就买了两套粉红色的小衣服，呵呵 男宝宝穿红色衣服也很可爱哦 ）。

  医生拿来一个仪器，接到言娃娃妈肚皮上，这个仪器可以打印出腹部压力的大小，伴随着每一次疼痛的开始，都会看到这个曲线向上走，疼痛缓解后曲线下降曲于平稳，一些具体的参数只有医生才看得明白，我们这些门外汗只能看个大概。

  到了晚上，终于有了一个床位，大房间，4张床的，这个时候已经顾不了这么多条件了，有张床已经非常不错了。晚上言娃娃奶奶送来鸡汤（我们到农村去买了8只土鸡），言娃娃妈妈非常的不舒服，只吃了一点点，看得人好心疼，真想下辈子我来变女人，让我来承受这些痛苦吧，哎。想到真真的疼痛还在后头，想起都害怕（先前我们了解过，生产的疼痛级别将达到12级，几乎接近人类的极限，当初陪言娃娃妈练习的拉玛泽呼吸法，不晓得生产的时候用得到不，希望有用）。

  晚上，又剩下我和言娃娃妈两个人了。等待，等待着什么时候可以上产房，我能做的就是每次疼痛到来时鼓励一下言娃娃妈，希望打印纸带上的曲线快点降下去。对面病房的来了一个新的产妇，一来就上了产房，两个小时后顺产了一个胖小子，真的是好羡慕她们，不晓得我们要等到什么时候去哦。

  就这样，等待着，等待着医生说可以上产房。中间有一次，言娃娃妈妈实在受住了，让我去问医生来打止痛针，我找到医生又被批了回来，看来这种情况医生见得多了，要想等到生产的时机，必须得经历这个疼痛的过程（当然，选择破腹产另当别论，现在的医院都是鼓励自己生产的）。疼痛产生尿频反映，每过二十钟我就和扶言娃娃妈去卫生间一次，然后在走道中走一走，分散注意力。

  晚上大概在3、4点钟的时候，我昏昏地睡去了，时不时听到言娃娃妈的痛苦声。后来听说言娃娃妈坚持不住了，自己出去找了医生，我真的是有些太不负责任了，呵呵，居然不知道有这事。迷糊中，听到来检查的护士说可以进产房了，我立马清醒过来，赶紧收拾东西（当时我们带来的所有东西，只有两条毛巾），跟着推车和言娃娃妈上了五楼，看了下表，这时是凌晨5点20分。

  上去后，我就只有在外面等（陪同人员一律不准进产房）。等候区的长登上坐着好几个在等的人，都时不时去门口打探一下。有一个妈妈迟迟不能生产，医生让她到外面来走走，我在外面等待的时候，只听到这个妈妈在走道里痛苦地呻吟，痛得腰都直不起来，变女人真恐怖。

  言娃娃妈进去后，还出来过一次，如果上了产床，就暂时见不到了她了。言娃娃妈进去后，很长时间没有出来，凌晨6点45分，我的手机收到一条短信：“破羊水了，不能走动了”。看来一会就可以生产了，破羊水在医院是非常安全的，如果发生在自己家中，就危险了。这时，我们俩的联系方式就只能靠手机短信了。

  外面的天还已有微微的泛白，我不时走到产房区门口想探出点情况。每次都看到忙碌的护士，有一个护士手中端着一个盆，里面全是血块，好血腥的。回到大厅等待，看到一波的医生出来了，另一班换班医生又进去，整个忙得不可开交。这时，我给言娃娃奶奶打了个电话，让她准备鸡汤。

  过了一阵，一个医生出来找我，说言娃娃妈想吃早点，但家里的还没准备好，再说了，要送过来也要一个小时，我只好下楼到永和买了豆桨和油条，找医生带了进去。

  我迫切想知道里面的情况，但却没有音讯，连短信都没有了，看来言娃娃妈开始了一个人的战斗，我真的无能为力，帮不上忙，心里干着急。

  早上10点多，言娃娃舅妈和奶奶来了，还带来了早餐，但我们这时候都只有在外面等。言娃娃舅妈等不及了，去门口找了个医生问了一下，医生说：“生了一个小时了，病人想吃荷包蛋，你们还没准备好哇？”我们被这一突来的消息感觉诧异，但更多的是喜悦，战斗结束了，我们胜利了，悬着的心终于放下来了。时间：早上11点45分，我们得知母子平安，后来从医生填写的记录，出生日期是早上10点33分，重量3350G，我们整整晚知道消息1个小时。

  言娃娃舅妈去楼下买了荷包蛋，送进去后（后来据言娃娃妈说，这是她吃过的最好吃的一顿饭），我们又等了半个小时左右，医生把言娃娃妈推出来了，一脸苍白，我第一时间跑上去，看到老婆留着喜悦的泪水对我微笑看得心里面好疼。只有用鼓励和感谢的抚摸安慰她。言娃娃就放在言娃娃妈妈腿上（后来据言娃娃妈说，他的重量好像一座大山，好重好重），用医院的一块军绿色包被包着。

  再看言娃娃，非常的安静，头上还有些许血丝，看得出挣扎着来到这个世界的努力。回到楼下病房，一切安排妥当，我去看言娃娃，没想到这个小崽崽也睁开了小眼睛，还在偷偷地看我呢，小手还在动，非常的可爱。言娃娃奶奶的脸像一朵绽开的花，说不出的高兴。

  当天晚上，奶奶陪我们一起在医院睡，言娃的外公在家里守家，并且也没闲着 外公忙着把土鸡拿去点杀后保鲜放到冰箱里、等着喂奶的言娃妈妈坐月子吃呢。真可谓一家老小齐上阵啊！呵呵 ！ 在医院睡觉的我们把言娃娃放到最里面，生怕被人换走，奶奶几乎一晚上没怎么睡，一直守在娃娃的小床边上呢。呵呵。有意思吧！整个晚上我们都没怎么睡好，病房太热，还有蚊子，而且不时有小孩醒来哭着要吃奶。由于医院床位紧张，9号下等我们就被医生“赶”回家了。

  现在的言娃娃，已经会喊爸爸了，并且即将学会走路，真的是一天一个样，我们虽然有些烦恼（我这一回家就带出去到外面牵着手走），但更多的是快乐，一种血的亲情，一种深深的缘分！

Jta事务在轻量级框架中的实现

2011年06月08日
